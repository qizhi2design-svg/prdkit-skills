# prdkit-prototype-update-page Skill

## 概述

当用户通过 viewer 点击页面元素并提供元素信息时，使用此 skill 进行精准修改。

## 触发场景

用户提供以下格式的元素信息时触发：

```
项目: 测试项目
文件: /Users/xxx/workspace/prototypes/mobile-demo/index.html
文件名: index.html

元素选择器: button#heroAction.primary-cta
标签: <button>
ID: heroAction
类名: primary-cta
文本内容: 立即体验

HTML:
<button class="primary-cta" id="heroAction">立即体验</button>
```

## 工作流程

### 第一阶段：解析元素信息
1. 解析用户提供的元素信息
2. 定位目标文件（index.html、script.js、mock.js）
3. 分析当前实现（HTML 结构、交互逻辑、关联数据）

### 第二阶段：明确修改意图并执行
4. 使用 `AskUserQuestion` 工具结构化提问
5. 明确四个维度：
   - **修改操作**：新增/修改/移除/替换？
   - **数据层面**：是否需要修改 mock.js？
   - **交互层面**：是否需要修改 script.js？
   - **视图层面**：是否需要修改 HTML 结构或样式？
6. 根据用户回答直接执行修改（数据 → 逻辑 → 视图）
7. 创建 checkpoint 版本
8. 检查服务状态并引导预览

## 核心特性

### 1. 精准定位
- 基于元素选择器快速定位代码位置
- 分析元素的 HTML、交互和数据关联

### 2. 结构化提问
- 四个维度明确修改意图：操作类型/数据/交互/视图
- 提供具体选项而非开放式问题
- 一次提问覆盖 2-3 个关键点

### 3. 快速执行
- 提问后直接执行修改，无需生成计划文件
- 按照数据 → 逻辑 → 视图的顺序修改
- 保持代码风格一致

### 4. 版本管理
- 修改完成后自动创建 checkpoint
- checkpoint 消息简短描述修改内容
- 支持修改前创建备份

### 5. 智能预览
- 检查服务状态
- 有服务运行 → 引导刷新浏览器
- 无服务运行 → 启动服务后引导访问

## 修改原则

### 修改操作类型

#### 新增元素
- 在合适位置插入新 HTML 元素
- 在 mock.js 添加相关数据
- 在 script.js 添加事件监听和交互逻辑
- 保持样式一致性

#### 修改元素
- 更新文本内容、属性或样式
- 同步更新 mock.js 数据
- 调整 script.js 逻辑
- 保持语义化和可访问性

#### 移除元素
- 从 HTML 删除目标元素
- 清理 mock.js 中不再使用的数据
- 移除 script.js 中相关的事件监听器
- 确保不影响其他功能

#### 替换元素
- 用新元素替换旧元素
- 更新 mock.js 数据结构
- 重写或调整 script.js 逻辑
- 保持页面整体一致性

### 代码层面原则

### 数据层（mock.js）
- 保持数据结构一致性
- 新增字段有合理默认值
- 状态数据存储在 `mockData.state`

### 逻辑层（script.js）
- 事件监听器命名清晰
- 状态更新后调用 `render()`
- 避免直接操作 DOM

### 视图层（index.html）
- 保持语义化 HTML
- ID 用于唯一元素，class 用于样式
- 图片占位使用纯色背景 + 文字

## 使用示例

### 示例 1：修改按钮文字和行为

**用户输入**：
```
元素选择器: button#heroAction
文本内容: 立即体验
```

**Skill 提问**：
```
关于「立即体验」按钮的修改：

1. 按钮文字是否需要修改？改为什么？
2. 点击后的行为：
   - A) 跳转到新页面
   - B) 打开弹窗
   - C) 触发动画
3. 是否需要添加加载状态？
```

**用户回答后直接执行修改**：
- 修改 index.html 按钮文字
- 修改 script.js 添加点击事件
- 创建 checkpoint
- 引导预览

### 示例 2：添加新功能

**用户输入**：
```
元素选择器: div.product-list
希望添加筛选功能
```

**Skill 提问**：
```
关于商品列表的筛选功能：

1. 筛选维度：
   - 价格范围？
   - 商品分类？
   - 其他条件？
2. 筛选方式：
   - 下拉选择？
   - 多选框？
   - 滑块？
3. 筛选后的行为：
   - 实时过滤？
   - 点击按钮后过滤？
```

**用户回答后直接执行修改**：
- 修改 mock.js 添加筛选数据
- 修改 script.js 添加过滤逻辑
- 修改 index.html 添加筛选器 UI
- 创建 checkpoint
- 引导预览

## 与 create-page skill 的区别

| 维度 | create-page | update-page |
|------|-------------|-------------|
| 触发时机 | 创建新页面 | 修改现有页面 |
| 输入信息 | 页面需求描述 | 元素选择器 + 修改需求 |
| 工作重点 | 整体布局和结构 | 精准定位和修改 |
| 提问方式 | 宽泛的需求澄清 | 聚焦的修改意图 |
| 工作流程 | 提问 → 生成计划 → 执行 | 提问 → 直接执行 |
| 输出产物 | 完整的页面文件 | 修改后的代码 + checkpoint |

## 最佳实践

1. **先理解后修改**：充分分析现有代码再提问
2. **小步快跑**：一次修改一个功能点
3. **保持一致**：遵循现有代码风格
4. **及时验证**：修改后立即测试
5. **版本管理**：重要修改前创建备份

## 注意事项

1. 不要跳过提问阶段直接修改
2. 提问后根据用户回答直接执行，无需生成计划文件
3. 不要一次修改过多内容
4. 不要破坏现有功能
5. 不要使用真实图片链接
6. 修改完成后必须创建 checkpoint
