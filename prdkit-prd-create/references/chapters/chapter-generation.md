# 章节级生成策略

## 通用规则

- 章节优先结构化输出，少写大段空泛描述
- 信息不足时标注 `[TODO: ...]`
- 推荐使用表格、列表、状态说明，而不是只写概述

## 章节建议

### 背景与问题

- 说明当前业务现状、核心问题、为什么现在要做
- 优先给出问题优先级

### 目标与成功标准

- 区分交付验收标准和上线成功标准
- 目标尽量量化

### 用户/角色与场景

- 区分需求提出人、直接使用人、受影响角色
- 至少写 1-2 个关键业务场景

### 范围（In/Out）

- 明确这次做什么、不做什么
- 范围边界比“尽量全”更重要

### 功能需求

- 按业务模块拆分
- 每个模块尽量包含目标、规则、交互、异常说明

### 业务规则 / 流程

- L2 可用规则列表
- L3/L4 应补充完整流程和关键分支

### 数据 / 埋点 / 指标

- 至少说明关键数据来源、关键事件、核心效果指标

### 权限 / 角色设计

- L3 写受影响角色
- L4 写角色矩阵、数据边界和权限差异

### 风险与依赖

- 区分产品风险、技术风险、运营风险、合规风险
- 每个高风险点尽量给出应对方式

### 里程碑 / 上线与回滚

- L3 至少有评审、开发、联调、上线节点
- L4 增加灰度、培训、回滚和推广策略
