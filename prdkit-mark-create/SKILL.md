# 创建原型标注

## 工作流程

### 第一阶段：分析元素上下文并确认操作

1. **解析用户提供的元素信息**，通常包含：
   - 项目名称
   - 文件路径（原型目录的绝对路径）
   - DOM 路径（元素的 DOM 层级）
   - 元素选择器（可选）
   - **标记文件路径**（如果已有标注）

2. **提取关键信息**：
   - 从文件路径中提取原型的相对路径
   - 例如：`/Users/xxx/workspace/prototypes/车后本地生活门店知识库` → `车后本地生活门店知识库`
   - 路径格式：`workspace/prototypes/` 之后的部分
   - 判断是否已有标注（是否包含标记文件路径）

3. **读取并分析原型页面**：
   - 读取 HTML 文件，查看元素的上下文
   - 分析元素的类型和位置
   - 查看元素的内容和属性

4. **产品 UX 业务层面分析**：

   **A. 页面信息分析**：
   - 这是什么页面？（首页、列表页、详情页、表单页等）
   - 页面的核心功能是什么？
   - 页面的用户场景是什么？
   - 页面在整个产品流程中的位置
   
   **B. 元素业务功能分析**：
   - 该元素在页面中的作用是什么？
   - 该元素承载什么业务功能？（查询、筛选、提交、展示等）
   - 该元素与用户的交互目标是什么？
   - 该元素在用户任务流程中的位置
   
   **C. 数据流分析**：
   - 该元素涉及哪些数据？（输入数据、展示数据、操作数据）
   - 数据从哪里来？（用户输入、API 获取、本地存储等）
   - 数据到哪里去？（提交到后端、更新界面、触发其他操作）
   - 数据的状态变化流程
   
   **D. 用户体验分析**：
   - 用户在这个元素上的预期行为是什么？
   - 该元素的交互反馈是否清晰？
   - 是否有异常状态或边界情况需要考虑？
   - 该元素与周边元素的关系（依赖、联动、互斥等）

5. **获取现有标注列表**：
   ```bash
   prdkit mark list --prototype <原型路径> --json
   ```
   - 查看该原型下已有哪些标注
   - 分析是否有相关的标注（相同或相近的选择器）
   - 查看是否有相关业务功能的标注

6. **如果已有标注**（包含标记文件路径）：
   ```bash
   prdkit mark get <mark-id> --prototype <原型路径> --json
   ```
   - 读取现有标注的内容
   - 展示给用户当前的标注信息

7. **综合分析并展示建议**：

   向用户展示分析结果：
   
   **A. 页面和元素的业务分析**：
   - 页面类型和核心功能
   - 元素的业务作用和用户场景
   - 数据流向和状态变化
   - 用户体验要点
   
   **B. 现有标注情况**：
   - 该原型共有 X 个标注
   - 该元素已有标注 / 该元素尚无标注
   - 相关业务功能的标注（如果有）
   
   **C. 建议的操作**：
   - **创建新标注**：如果需要记录业务需求、交互说明、数据规则等
   - **修改现有标注**：如果元素已有标注，但业务需求或规则有变化
   - **修改页面代码**：如果是要改变元素本身（样式、内容、交互实现）
   - **不需要操作**：如果元素已有完整标注，且业务逻辑无变化

8. **使用 `AskUserQuestion` 工具确认操作**：

   **必须明确的维度**：
   - **操作类型**：你想做什么？
     - 创建新标注（记录业务需求、交互说明、数据规则等）
     - 修改现有标注（更新业务需求或规则）
     - 修改页面代码（改变元素的样式、内容或交互实现）
     - 暂不操作（仅查看分析结果）

9. **根据用户选择，决定下一步**：
   - 如果选择"创建新标注" → 继续第二阶段
   - 如果选择"修改现有标注" → 提示用户应使用 `/prdkit-prototype-update-mark`，结束流程
   - 如果选择"修改页面代码" → 提示用户应使用 `/prdkit-prototype-update-page`，结束流程
   - 如果选择"暂不操作" → 友好地结束流程

### 第二阶段：明确标注内容

9. **构建 CSS 选择器**：
   - 如果用户提供了完整的选择器，直接使用
   - 如果只有 DOM 路径，将其转换为 CSS 选择器格式
   - 例如：`body > div.app-shell > aside.sidebar > div.brand`

10. **使用 `AskUserQuestion` 工具结构化提问**，明确标注的目的和内容：

   **必须明确的维度**：
   - **标注类型**：这个标注是关于什么的？
     - 功能说明 说明
     - 交互行为说明
     - 数据展示说明
     - 其他
   
   - **标注标题**：用一句话概括这个标注的核心内容
   
   - **标注描述**：详细说明（支持 Markdown 格式）
     - 如果是 功能说明：说明功能需求、用户价值、验收标准
     - 如果是交互行为：说明交互流程、状态变化、边界情况
     - 如果是数据展示：说明数据来源、展示规则、异常处理
     - 如果是其他：根据实际情况提供相关说明

   提问要点：
   - 简短、具体、聚焦当前元素
   - 提供常见标注类型的选项
   - 引导用户提供结构化的描述

### 第三阶段：创建标注

11. **构建 prdkit mark create 命令**：
   ```bash
   prdkit mark create \
     --prototype <原型路径> \
     --title "<标注标题>" \
     --selector "<CSS选择器>" \
     --dom-path "<DOM路径>" \
     --desc "<标注描述>"
   ```

12. **执行命令创建标注**：
   - 运行 prdkit mark create 命令
   - 检查命令执行结果
   - 如果成功，提示用户标注已创建
   - 如果失败，分析错误原因并提示用户

13. **提示用户查看**：
   - 告知标注文件的位置
   - 提示可以在 viewer 中查看标注
   - 说明标注会自动创建 checkpoint

## 标注类型

提供三种核心标注类型，详见 `references/templates/` 目录：

- **功能说明**：说明产品功能需求
- **交互行为说明**：描述用户交互流程和系统响应
- **数据展示说明**：说明数据来源、展示规则和异常处理

## 约束规则

- 必须先解析元素信息，提取原型路径
- 必须使用 `AskUserQuestion` 工具明确标注内容
- 提供标注类型选项，引导用户选择
- 原型路径必须是相对路径（相对于 workspace/prototypes/）
- 创建标注后提示用户查看

## 最佳实践

1. **标题简洁明确**：用一句话说清楚标注的核心内容
2. **描述结构化**：使用 Markdown 格式，分段落、列表组织内容
3. **聚焦当前元素**：标注内容应该与选中的元素直接相关
4. **提供上下文**：说明为什么需要这个标注，背景是什么

## 示例

**用户输入**：
```
项目: 测试项目
文件路径: /Users/xxx/workspace/prototypes/车后本地生活门店知识库
DOM 路径: body > div.app-shell > aside.sidebar > div.brand
```

**Skill 提问**：
```
关于这个元素的标注：

1. 标注类型：
   - A) 功能说明
   - B) 交互行为说明
   - C) 数据展示说明
   - D) 其他

2. 标注标题（一句话概括）：

3. 标注描述（详细说明）：
```

**用户回答**：
- 类型：A) 功能说明
- 标题：品牌展示区
- 描述：展示门店品牌信息，包括 logo 和名称

**执行命令**：
```bash
prdkit mark create \
  --prototype "车后本地生活门店知识库" \
  --title "品牌展示区" \
  --selector "body > div.app-shell > aside.sidebar > div.brand" \
  --dom-path "body > div.app-shell > aside.sidebar > div.brand" \
  --desc "## 功能说明
展示门店品牌信息，帮助用户识别当前门店

## 展示内容
- 门店 logo
- 门店名称

## 交互行为
点击品牌区可返回首页"
```

## 注意事项

- 确保正确提取原型的相对路径
- 引导用户提供有价值的标注内容
- 告知用户标注创建后会自动创建版本快照
