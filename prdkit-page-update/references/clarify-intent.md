# 明确修改意图

## 三个核心维度

在用户提供元素信息后，必须通过结构化提问明确以下三个维度：

### 1. 数据修改
- 是否需要修改 mock.js 中的数据？
- 需要新增、修改还是删除数据字段？
- 数据结构是否需要调整？

### 2. 交互逻辑
- 是否需要修改 script.js 中的事件处理？
- 点击/输入/滚动等交互行为是否需要改变？
- 状态管理逻辑是否需要调整？

### 3. 新增功能
- 是否需要添加新的 HTML 元素？
- 是否需要新增功能模块（如弹窗、表单、列表等）？
- 是否需要添加新的交互入口？

## 提问模板

### 单元素修改

当用户选中单个元素时，使用以下提问结构：

```
关于 [元素描述]（如「立即体验」按钮）的修改，请确认：

1. **文案和数据**：
   - 按钮文字是否需要修改？
   - 是否需要调整按钮的状态（禁用/加载中）？

2. **点击行为**：
   - 点击后应该发生什么？（跳转/弹窗/状态变化/数据提交）
   - 是否需要添加确认提示或加载动画？

3. **视觉和位置**：
   - 样式是否需要调整（颜色/大小/位置）？
   - 是否需要添加图标或其他装饰元素？
```

### 多元素修改

当用户选中多个元素时：

```
关于选中的 [N] 个元素，请确认修改范围：

1. **批量修改**：这些元素是否需要统一修改（如统一样式、统一行为）？

2. **关联修改**：这些元素之间是否有交互关联（如 tab 切换、单选组）？

3. **独立修改**：每个元素是否有各自的修改需求？请逐一说明。
```

### 功能性修改

当修改涉及新增功能时：

```
关于新增 [功能描述]，请确认：

1. **触发方式**：如何触发这个功能？（点击按钮/自动触发/条件触发）

2. **功能流程**：
   - 用户操作步骤是什么？
   - 成功/失败时分别如何反馈？

3. **数据需求**：
   - 需要展示哪些数据？
   - 数据从哪里来？（mock 数据/用户输入/计算结果）
```

## 提问技巧

### 1. 聚焦当前元素
- ✅ "这个按钮点击后应该跳转到哪个页面？"
- ❌ "整个页面的交互流程是什么？"

### 2. 具体可操作
- ✅ "按钮文字改为「开始使用」还是「立即注册」？"
- ❌ "按钮文案怎么改比较好？"

### 3. 提供选项
- ✅ "点击后是：A) 打开弹窗 B) 跳转新页面 C) 展开详情？"
- ❌ "点击后做什么？"

### 4. 分层提问
- 先问核心行为（做什么）
- 再问细节（怎么做）
- 最后问边界情况（异常处理）

## 常见场景的提问示例

### 场景 1：修改按钮

```
关于「立即体验」按钮的修改：

1. 按钮文字是否需要修改？如果需要，改为什么？

2. 点击后的行为：
   - A) 跳转到新页面（请提供页面名称）
   - B) 打开弹窗/抽屉（请描述弹窗内容）
   - C) 触发动画或状态变化（请描述效果）

3. 是否需要添加加载状态或点击反馈？
```

### 场景 2：修改列表项

```
关于列表项的修改：

1. **数据内容**：
   - 是否需要修改显示的字段（标题/描述/标签等）？
   - 是否需要调整数据的数量或顺序？

2. **交互行为**：
   - 点击列表项应该做什么？（查看详情/选中/编辑）
   - 是否需要添加操作按钮（删除/编辑/分享）？

3. **状态展示**：
   - 是否需要显示不同状态（已读/未读、完成/进行中）？
   - 空状态时如何展示？
```

### 场景 3：修改表单

```
关于表单的修改：

1. **字段调整**：
   - 需要新增、删除或修改哪些表单字段？
   - 哪些字段是必填的？

2. **验证规则**：
   - 各字段的验证规则是什么？（格式/长度/范围）
   - 验证失败时如何提示？

3. **提交行为**：
   - 提交后做什么？（显示成功提示/跳转/关闭弹窗）
   - 提交失败时如何处理？
```

### 场景 4：修改导航

```
关于导航的修改：

1. **导航项**：
   - 需要新增、删除或重命名哪些导航项？
   - 默认选中哪一项？

2. **切换行为**：
   - 切换导航时内容如何变化？（切换 tab/加载新数据/跳转页面）
   - 是否需要保留切换历史？

3. **视觉反馈**：
   - 当前选中项如何高亮？
   - 是否需要添加图标或徽章？
```

## 信息不足时的处理

### 用户回答不完整
- 做合理假设，但必须在计划中明确标注
- 记录到 `draft/reference/<page>-update-plan.md` 的「待确认项」

### 用户回答模糊
- 提供具体选项让用户选择
- 用示例说明不同选项的效果

### 用户回答矛盾
- 指出矛盾点，请用户澄清
- 提供建议方案供参考

## 提问后的输出

提问完成后，将确认的信息整理为结构化记录：

```markdown
## 修改意图确认

### 目标元素
- 选择器：`button#heroAction.primary-cta`
- 当前文本：「立即体验」
- 当前行为：无交互

### 修改需求

**数据修改**
- 按钮文字改为「开始使用」
- 无需修改 mock.js

**交互逻辑**
- 点击后跳转到「功能介绍」页面
- 添加点击反馈动画

**新增功能**
- 无需新增功能

### 待确认项
- 跳转动画效果（假设：淡入淡出）
```

这个记录将作为改造计划的基础。
