# 更新需求标注

修改原型上已有标注的标题或说明内容，适应需求变化。

## 核心原则

- **纯产品视角**：全程使用产品经理的语言。描述功能时只关注"用户能做什么"、"系统如何响应"，不描述"界面长什么样"。禁止出现 UI 组件名称（按钮、输入框、弹窗、抽屉、标签页、卡片、列表等）、样式属性（颜色、宽度、高度、间距、字体等）和技术实现词汇（CSS 选择器、DOM 路径、API 调用、前端框架等）
- **结果导向**：不展示命令构造过程，不让用户看到技术实现细节。所有命令在后台执行，用户只看到结果
- **不引外部**：最终输出和用户沟通中不引用外部文档或链接，不出现 URL 地址、项目文档引用等
- **一次成功**：在运行编辑命令前确保所有参数完整准确，不在用户面前试错

## 工作流程

### 第一步：确定原型和标注

**确定原型路径：**

如果产品经理没有明确说明修改哪个原型，先询问：
- "您想修改哪个原型的标注？"
- 用简单的方式描述，如"需求管理后台"、"Dashboard"、"用户中心"

**列出该原型的所有标注，用产品语言展示：**

```bash
prdkit prototype mark list --prototype "<原型相对路径>"
```

展示时只显示标题和简要说明，不展示标注 ID、选择器、DOM 路径等技术信息：

```
该原型有以下标注：

1. [标题 1] — [一句话概括]
2. [标题 2] — [一句话概括]
3. [标题 3] — [一句话概括]
```

询问："您想修改哪个标注？"

### 第二步：查看标注当前内容

获取选中的标注详情（在后台完成，不展示命令）：

```bash
prdkit prototype mark get <标注ID> --prototype "<原型相对路径>"
```

向产品经理展示当前内容。**注意：如果标注原文包含组件名称（按钮、输入框等）或技术描述（颜色、宽度等），要先转换为产品语言再展示。**

```
当前标注 — [标题]

[当前描述内容，已转换为纯产品语言]
```

### 第三步：收集修改内容

用简单的语言询问修改需求，不要问"变更类型"、"影响范围"等抽象概念：

**引导性问题：**

- "您想怎么改？"
- "是改标题、改说明内容，还是两个都改？"

**常见的修改场景：**

- **需求调整**："需求变了，说明要跟着改" → 补充或替换内容
- **补充细节**："写得不够清楚，再补充一下" → 在原内容基础上追加
- **修正错误**："这里写错了" → 修正指定部分
- **精简内容**："太啰嗦了，精简一下" → 重写或删减

如果产品经理不完全清楚改什么，可以参考这些角度提供建议：
- 功能说明类标注：功能的范围和边界是否需要调整
- 交互行为类标注：操作流程或系统响应是否有变化
- 数据展示类标注：数据字段或展示规则是否更新

**确认修改：**

展示修改后的效果（标题变更直接展示，描述变更展示修改前后对比）。**确保修改后的描述使用纯产品语言，不出现组件名称、样式描述或技术术语。**

```
修改前：「原标题」
修改后：「新标题」

说明内容：
[修改后的描述全文，纯产品语言]
```

询问："这样修改可以吗？"

如果涉及多行内容修改，直接和产品经理确认修改后的完整内容，不要分段确认。

### 第四步：更新标注

确认无误后，在后台执行更新。

**只修改标题：**

```bash
prdkit prototype mark edit <标注ID> \
  --prototype "<原型相对路径>" \
  --title "<新标题>"
```

**同时修改标题和说明（推荐使用文件方式，避免命令行换行问题）：**

```bash
# 将完整描述写入临时文件
cat > /tmp/prdkit-mark-desc.md << 'MARKDOWN'
[修改后的完整说明内容，包含 Markdown 格式]
MARKDOWN

# 通过 desc-file 参数更新标注
prdkit prototype mark edit <标注ID> \
  --prototype "<原型相对路径>" \
  --title "<新标题>" \
  --desc-file /tmp/prdkit-mark-desc.md
```

**只修改说明内容：**

```bash
cat > /tmp/prdkit-mark-desc.md << 'MARKDOWN'
[新的说明内容]
MARKDOWN

prdkit prototype mark edit <标注ID> \
  --prototype "<原型相对路径>" \
  --desc-file /tmp/prdkit-mark-desc.md
```

**简短说明可以直接内联：**

```bash
prdkit prototype mark edit <标注ID> \
  --prototype "<原型相对路径>" \
  --desc "<新的简短说明>"
```

**关键校验：**
- 标注 ID 必须正确（从第一步的 list 结果中获取）
- 原型路径是相对路径（相对于 `workspace/prototypes/`）
- 至少提供 --title 或 --desc/--desc-file 中的一个
- --desc 和 --desc-file 不能同时使用
- 多行内容必须使用 --desc-file 写入临时文件，不能直接写内联的 `\n`

### 第五步：告知结果

用产品语言告知修改完成：

```
已更新「[标注标题]」的标注内容。
```

如果只修改了标题或说明，分别说明：

```
已更新「[标注标题]」的标注内容。
- 标题已修改
- 说明内容已更新
```

不需要提及：
- 标注文件的路径和位置
- 标注 ID 等内部标识
- 命令执行的具体参数
- checkpoint 创建等内部机制

如果更新失败，直接输出错误信息，不要做技术分析。

## 注意事项

### 产品语言红线（禁止出现）

以下词汇在任何输出中都**禁止出现**，包括展示标注内容、确认修改、告知结果等环节：

| 类别 | 禁止词汇 | 应替换为 |
|------|----------|----------|
| 组件名 | 按钮、输入框、弹窗、抽屉、标签页、卡片、列表、下拉框、单选、复选、开关、滑块、分页器、面包屑、步骤条、树形控件、日期选择器、穿梭框、上传组件 | 用功能描述代替，如"点击此处可以…" |
| 样式属性 | 颜色名（红色、橙色、蓝色、白色等）、宽度、高度、间距、边距、字体、字号、行高、圆角、阴影、边框、背景色、透明度 | 去掉，不描述视觉样式 |
| 技术实现 | CSS 选择器、DOM 路径、API 调用、前端框架、组件库、Vue、React、HTML、class、id、div、span、flex、grid、z-index | 完全禁止出现 |
| 内部标识 | 标注 ID、标注文件名、mark-xxx、文件路径、workspace/prototypes | 完全禁止出现 |
| 工具术语 | 命令、命令行、参数、终端、bash、`$` 符号 | 完全禁止出现 |

### 内容转换规则

当读取标注的原始内容时，如果原文包含上述禁止词汇，**必须自动转换为产品语言后再展示给用户**：

- ❌ "点击黄色按钮后右侧出现drawer" → ✅ "点击确认后，右侧会展开详情面板"
- ❌ "首屏顶部tab栏有3个标签页" → ✅ "首屏顶部提供三个切换入口"
- ❌ "列表每行右侧有操作按钮" → ✅ "每条数据都配有对应的操作入口"

### 描述编写规范

- **只写"做什么"和"为什么"**，不写"怎么做"
  - ✅ "审批通过后，系统自动通知下一节点处理人"
  - ❌ "点击绿色按钮调用审批接口，通过后发送通知"
- **用功能描述代替组件名称**
  - ✅ "用户在此处查看订单状态和物流信息"
  - ❌ "卡片列表展示订单信息和物流状态"
- **用业务语言代替技术语言**
  - ✅ "支持按创建时间、优先级排序"
  - ❌ "表格支持创建时间和优先级列的排序功能"
- **关注交互目的而非交互形式**
  - ✅ "从备选列表中选中需要分配的人员"
  - ❌ "左边是待选列表右边是已选列表，中间有穿梭箭头"

### 修改影响提醒

- **标题修改影响识别**：标题用于列表展示和 PRD 引用，修改后会影响已在 PRD 中引用的内容。如果确认修改标题，检查是否需要同步更新 PRD 中的引用
- **标注粒度保持**：一个标注只描述一个独立的功能点或需求。如果产品经理想在一个标注里塞多个不相关需求，建议拆成多个标注（使用 prdkit-mark-create）
- **废弃需求处理**：如果需求被废弃，建议删除该标注而非将内容留空，以免给后续阅读者造成困惑
- **PRD 同步提醒**：如果此标注已通过 prdkit-prototype-sync 同步到 PRD，修改后建议重新同步，保持 PRD 与原型一致

### 与产品经理沟通要点

- **修改前必须展示当前内容**，让产品经理清楚现在是什么样再决定怎么改
- **展示列表时隐藏技术细节**，只展示标题和简述，不展示 ID、选择器、DOM 路径
- **修改前后对比展示**，让产品经理确认改动是否符合预期
- **不要问抽象概念**，用"您想怎么改"代替"变更类型是什么"、"影响范围是什么"
- **提供修改建议**，而不是让产品经理从零开始描述
