# PRD 审查路径

## 1. 审查前先判断四件事

1. 商业属性：商业化产品 / 企业自研系统
2. 功能类型：业务型管理软件 / 工具型软件 / 交易型平台 / 基础服务型
3. 文档范围：0-1 系统级规划 / 迭代需求 / 模块级需求
4. 是否涉及 AI 功能

## 2. 维度适用性

| 维度 | 适用条件 | 不适用时处理 |
| --- | --- | --- |
| 07 数据建模 | 业务型管理软件、交易型平台 | 标注不适用，仅检查关键数据结构 |
| 10 商业分析 | 商业化产品 | 自研系统改看 ROI / 投入产出 |
| 03 竞品相关 | 商业化产品 | 自研系统改看同类系统参考 |
| 06 企业架构深层检查 | 0-1 / 系统级规划 | 迭代需求只看变更架构边界 |
| 13 AI 功能 | 文档涉及 AI | 标注不适用 |
| R8 多租户风险 | 商业化 SaaS | 标注不适用 |

## 3. 优先级定义

- P0：影响核心业务闭环、合规、安全或基础定位
- P1：重要场景上线前应修正
- P2：明显改进项，但不阻塞推进
- P3：优化建议

## 4. 评级建议

- 优秀：关键证据完整，方法和落地细节都比较扎实
- 合格：核心内容具备，但仍有少量边界和细节欠缺
- 待改进：主要结构在，但存在明显漏项或证据不足
- 严重缺失：关键章节、关键流程或关键约束缺失

## 5. 评审风格

- 先看证据，再下判断
- 先给可执行建议，再给抽象评价
- 不以 L4 的要求去苛责 L1 / L2
- 自研和商业化用不同视角，避免误判
