# 需求描述回写约定

## 核心原则

- `feature list` 不是固定模块
- prototype 反哺 PRD 时，优先写回“现有的具体需求描述落点”
- 先找章节归属，再决定怎么写

## 优先落点顺序

优先按以下顺序判断本次内容应写到哪里：

1. `## 7. 功能需求` 下已存在的页面 / 模块 / 功能小节
2. `## 8. 业务规则 / 流程` 下对应规则说明
3. `## 9. 交互与内容要求` 下对应交互说明
4. `## 15. 待确认 / 开放问题` 下对应未决事项
5. 如果以上都没有合适落点，再在 `## 7. 功能需求` 下新增对应页面 / 模块小节

## 推荐识别方式

### 已有章节可直接复用的信号

- 标题里已经出现相同页面名、模块名、业务对象名
- 小节内容和当前 prototype 的页面目标一致
- 该小节本来就在描述对应功能，而不是更上层的背景或目标

### 不应写入的区域

- `背景与问题`
- `目标与成功标准`
- `里程碑`
- 已经明确是历史修订内容的旧记录

## 回写结构建议

如果目标章节已有对应小节：

- 保留原有小节结构
- 在其中增量更新：
  - 功能描述
  - 交互说明
  - 规则约束
  - 待确认项

如果目标章节没有对应小节：

```md
### <页面 / 模块名称>

- Prototype Path: `<prototype-path>`
- Prototype Link: `<link>`
- Sync Source: `<mark-count> marks`

#### 需求描述
- ...

#### 交互要求
- ...

#### 业务规则 / 约束
- ...

#### 待确认项
- ...
```

## mark 到 PRD 的映射建议

- 页面定位、区域职责、主任务 -> `需求描述`
- 按钮、表单、状态、反馈、跳转 -> `交互要求`
- 条件、边界、限制、角色差异 -> `业务规则 / 约束`
- 信息不足、描述冲突、开放判断 -> `待确认项`

## 差异识别

至少识别三类变化：

- 新增：PRD 原来没有，本次 prototype 新沉淀出来
- 更新：PRD 原来有，但本次被细化、修正或重写
- 删除：PRD 原来有，但当前 prototype / marks 已不再支持该描述

## 写法要求

- 不要把 mark 原文整段贴回 PRD
- 不要为了同步而新造一个“Feature List”章节
- 优先让 PRD 看起来像自然演进后的需求文档，而不是外挂一层同步结果
