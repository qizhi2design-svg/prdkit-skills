# 需求描述回写约定

## 核心原则

- `feature list` 不是固定模块
- prototype 反哺 PRD 时，优先写回“现有的具体需求描述落点”
- 先找章节归属，再决定怎么写

## 优先落点顺序

优先按以下顺序判断本次内容应写到哪里：

1. `## 7. 功能需求` 下已存在的页面 / 模块 / 功能小节
2. `## 8. 业务规则 / 流程` 下对应规则说明
3. `## 9. 交互与内容要求` 下对应操作流程说明
4. `## 15. 待确认 / 开放问题` 下对应未决事项
5. 如果以上都没有合适落点，再在 `## 7. 功能需求` 下新增对应页面 / 模块小节

## 推荐识别方式

### 已有章节可直接复用的信号

- 标题里已经出现相同页面名、模块名、业务对象名
- 小节内容和当前 prototype 的页面目标一致
- 该小节本来就在描述对应功能，而不是更上层的背景或目标

### 不应写入的区域

- `背景与问题`
- `目标与成功标准`
- `里程碑`
- 已经明确是历史修订内容的旧记录

## 回写结构建议

如果目标章节已有对应小节：

- 保留原有小节结构
- 在其中增量更新：
  - 功能描述
  - 操作流程
  - 规则约束
  - 待确认事项

如果目标章节没有对应小节：

```md
### <页面 / 模块名称>

- Prototype Path: `<prototype-path>`
- Prototype Link: `<cloud-url | local-url | relative-path>`（按优先级取）
- Sync Source: `<mark-count> marks`

#### 功能说明
- ...

#### 操作流程
- ...

#### 业务约束
- ...

#### 待确认事项
- ...
```

> Prototype Link 统一使用云端 release URL，从 `prototype_release_link_list` 获取

## 标注到 PRD 的映射建议

- 页面定位、区域职责、主任务 -> `功能说明`
- 操作入口、信息录入、状态变化、系统反馈、页面导航 -> `操作流程`
- 条件、边界、限制、角色差异 -> `业务约束`
- 信息不足、描述冲突、开放判断 -> `待确认事项`

## 差异识别

至少识别三类变化：

- 新增：PRD 原来没有，本次 prototype 新沉淀出来
- 更新：PRD 原来有，但本次被细化、修正或重写
- 删除：PRD 原来有，但当前 prototype / marks 已不再支持该描述

## 指标过滤规则

标注中可能包含交互指标或技术指标，同步到 PRD 时需要按以下规则处理：

### 可转换的指标

| 标注中的原始指标 | 转换为 | 写入位置 |
|------------------|--------|----------|
| 点击率 / 点击量 | 功能使用率 / 功能使用频次 | 功能说明 |
| 转化率 | 任务完成率 | 功能说明 / 操作流程 |
| 页面停留时长 | 任务处理效率 | 操作流程 |
| 加载时间 / 响应时长 | 系统响应效率 | 功能说明（仅当有明确业务要求时） |
| PV / UV | 功能覆盖范围 / 用户触达率 | 功能说明 |

### 应过滤不同步的指标

以下类型的指标如果没有对应的产品/业务指标可以转换，应直接丢弃，不同步到 PRD：

- 纯前端性能指标（首屏渲染时间、页面加载时间、资源加载大小等）
- 技术架构指标（接口并发数、数据库查询量、缓存命中率等）
- 设计验证指标（用户测试完成率、设计满意度评分等）

### 转换原则

1. **能转换的必须转换**：标注原文包含交互指标时，先判断是否有对应的产品/业务指标可以表达相同意图
2. **不能转换的丢弃**：没有对应产品/业务指标的交互/技术指标，不写入 PRD
3. **转换后放在对应类别下**：转换后的指标写入对应的功能说明或操作流程类别，不独立成节
