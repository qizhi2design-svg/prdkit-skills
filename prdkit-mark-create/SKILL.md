# 创建需求标注

为原型中的页面元素添加需求标注，记录产品功能、交互行为或数据展示规则。

## 核心原则

- **纯产品视角**：全程使用产品经理的语言。描述功能时只关注"用户能做什么"、"系统如何响应"，不描述"界面长什么样"。禁止出现 UI 组件名称（按钮、输入框、弹窗、抽屉、标签页、卡片、下拉菜单、列表等）、样式属性（颜色、宽度、高度、间距、字体等）和技术实现词汇（CSS 选择器、DOM 路径、API 调用、前端框架等）
- **结果导向**：不展示命令构造过程，不让用户看到技术实现细节
- **不引外部**：最终输出和用户沟通中不引用外部文档或链接，不出现 URL 地址、项目文档引用等
- **一次成功**：在运行命令前确保所有参数完整准确

## 工作流程

### 第一步：解析元素信息

用户从 viewer 复制的内容通常包含：

- **项目名称**：如 `项目: 电商后台`
- **文件路径**：包含原型目录的完整路径
- **页面元素**：用户选中的页面区域

从中提取关键信息：

1. **原型路径**：从 `文件路径` 中提取 `workspace/prototypes/` 之后的相对路径
   - 如 `/Users/xxx/workspace/prototypes/车后门店/首页` → `车后门店/首页`
   - 如 `/Users/xxx/workspace/prototypes/dashboard` → `dashboard`
2. **项目名称**：直接使用用户提供的项目名
3. **页面元素标识**：从元素信息中识别是哪个页面区域（如搜索区、筛选条件、数据列表、操作入口等），用功能描述代替组件名称

### 第二步：确认操作意图

如果用户意图不明确，用 `AskUserQuestion` 简短确认：

```
我看到你选中了 [页面区域描述]，需要为这个区域添加需求标注吗？
```

如果用户已明确说要"创建标注"或"标注一下"，直接进入第三步。

如果该区域已有标注，先展示现有标注，再询问是否需要新增或修改。

### 第三步：收集标注内容

通过结构化提问收集标注信息。每次只问一个主题，避免一次性抛出多个问题：

**1. 标注类型**（单选）：
- A) 功能说明 — 这个区域提供什么功能、解决什么问题
- B) 交互行为 — 用户怎么操作、系统如何响应
- C) 数据展示 — 数据来源、展示格式、异常处理

**2. 标注标题** — 用一句话概括（5-15 字），如"商品搜索功能"、"注册表单校验规则"

**3. 标注描述** — 根据标注类型，参考对应的模板（详见 `references/templates/`）与用户沟通并组织内容：

- **功能说明**：`references/templates/prd-feature.md`
- **交互行为**：`references/templates/interaction.md`
- **数据展示**：`references/templates/data-display.md`

每种模板都提供了基础结构和详细结构。根据用户提供的信息完整度选择合适的详细程度，不需要逐项询问所有字段。

### 第四步：创建标注

**在运行命令前，确保以下信息完整：**

1. `--prototype`：从文件路径中提取的原型相对路径（必须正确）
2. `--title`：用户确认的标注标题
3. `--selector`：从用户提供的元素信息中提取的选择器
4. `--desc`：包含 Markdown 格式的描述内容

**描述内容写入临时文件（推荐）：**
多行 Markdown 描述应先写入临时文件，再通过 `--desc-file` 传入，避免命令行换行符转义问题：

```bash
# 将描述写入临时文件
cat > /tmp/prdkit-mark-desc.md << 'MARKDOWN'
## 功能说明
- 目标用户：...
- 核心价值：...

## 交互流程
1. ...
MARKDOWN

# 通过 desc-file 参数创建标注
prdkit prototype mark create \
  --prototype "<原型相对路径>" \
  --title "<标注标题>" \
  --selector "<CSS选择器>" \
  --desc-file /tmp/prdkit-mark-desc.md
```

**简短的描述也可以内联传入：**

```bash
prdkit prototype mark create \
  --prototype "<原型相对路径>" \
  --title "<标注标题>" \
  --selector "<CSS选择器>" \
  --desc "<简短描述>"
```

**关键校验：**
- 原型路径是相对路径（相对于 `workspace/prototypes/`），不要包含完整路径
- 选择器不为空
- 标题不为空且有实际含义
- 描述内容使用正确的 Markdown 格式（`\n` 会被当做字面字符，多行内容必须用 `--desc-file`）

### 第五步：告知结果

标注创建成功后，用产品语言告知用户：

```
已为「[页面区域]」创建 [类型] 标注：**[标题]**
```

不需要提及：
- 标注文件的位置和文件名
- 命令执行的具体参数
- checkpoint 创建等内部机制

如果创建失败，直接输出错误信息，不要做技术分析。

## 标注内容编写指引

### 功能说明类

适合描述独立功能模块，关注业务价值。

- 标题示例："商品搜索功能"、"用户注册流程"、"权限管理模块"
- 描述重点：功能价值、操作步骤、预期结果、异常处理

### 交互行为类

适合描述用户与页面的互动流程。

- 标题示例："搜索筛选联动"、"拖拽排序调整"、"展开收起切换"
- 描述重点：触发方式、状态变化、反馈形式、边界情况

### 数据展示类

适合描述数据的内容、格式和展示规则。

- 标题示例："订单数据规则"、"用户信息展示"、"数据统计面板"
- 描述重点：数据来源、格式化规则、空状态/错误状态

## 注意事项

### 产品语言红线（禁止出现）

以下词汇在任何输出中都**禁止出现**，包括展示标注内容、与用户沟通等环节：

| 类别 | 禁止词汇 | 应替换为 |
|------|----------|----------|
| 组件名 | 按钮、输入框、弹窗、抽屉、标签页、卡片、下拉菜单、列表、单选、复选、开关、滑块、分页器、面包屑、步骤条、树形控件、日期选择器、穿梭框 | 用功能描述代替，如"点击此处可以…" |
| 样式属性 | 颜色名（红色、橙色、蓝色、白色等）、宽度、高度、间距、边距、字体、字号、行高、圆角、阴影、边框、背景色 | 去掉，不描述视觉样式 |
| 技术实现 | CSS 选择器、DOM 路径、API 调用、前端框架、组件库、Vue、React、HTML、class、id、div、span | 完全禁止出现 |
| 内部标识 | 标注 ID、标注文件名、mark-xxx、文件路径、workspace/prototypes | 完全禁止出现 |

### 标题编写规范

- **用功能描述代替组件名**
  - ✅ "商品搜索"、"筛选条件组合"、"排序切换"
  - ❌ "搜索按钮"、"下拉菜单"、"排序弹窗"
- **5-15 字**，让读者一眼看懂标注覆盖的内容
- **关注业务目的**，而不是交互形式

### 描述编写规范

- **只写"做什么"和"为什么"**，不写"怎么做"
  - ✅ "用户输入关键词后可快速定位到匹配的商品"
  - ❌ "点击搜索按钮调用搜索 API，返回结果渲染到列表"
- **用功能描述代替组件名称**
  - ✅ "在此区域选择筛选条件后，列表自动刷新"
  - ❌ "下拉框选择筛选条件后，表格重新加载"
- **关注用户目标和系统响应**
  - ✅ "提交后系统自动校验信息，校验通过进入下一步"
  - ❌ "点击提交按钮触发表单校验，通过后跳转到下一页"

### 与产品经理沟通要点

- **用结构化提问代替开放式问题**：给出选项让用户选择，而不是让用户从零开始想
- **确认页面区域**：展示用户选中区域的描述，确认是否理解正确
- **按类型引导**：根据标注类型（功能/交互/数据）有针对性地提问缺失的关键信息
- **不要问抽象概念**："标注的作用是什么" → "这个区域主要解决什么问题"

### 标注粒度原则

- **一个标注只描述一个功能点**：不要在一个标注里塞多个不相关的需求
- **如果区域包含多个独立功能**：建议创建多个标注
- **如果功能跨多个区域**：在主要区域创建标注，在其他区域添加引用说明
